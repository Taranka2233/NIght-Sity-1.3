# Развёртывание Найт-Сити Нет 2.077.206

Развёртывание состоит из трёх независимых частей: Firebase, Cloudflare Worker и Android APK. Выполняйте их в указанном порядке.

## 1. Firebase

Проект должен соответствовать `project_id` из `google-services.json`, Android package — `net.nightcity.chat`.

```bash
firebase deploy --only firestore:rules,storage
```

Также включите в Firebase Authentication вход по email/паролю. Firestore должен быть создан в production mode.

Важно: `storage.rules` намеренно запрещает все операции. В текущей версии небольшие E2E-вложения передаются в зашифрованном сообщении; открытая загрузка файлов в Storage не допускается.

## 2. Cloudflare Worker

Разверните `worker.js` как module Worker. Задайте секреты:

```bash
wrangler secret put FB_CLIENT_EMAIL
wrangler secret put FB_PRIVATE_KEY
```

И обычные переменные:

- `FB_PROJECT_ID` — Firebase project id;
- `ALLOWED_ORIGINS` — список допустимых origin через запятую, минимум `https://localhost,capacitor://localhost`.

Сервисному аккаунту нужны минимальные права на чтение документов Firestore и отправку Firebase Cloud Messaging. Не помещайте JSON-ключ сервисного аккаунта, приватный ключ или release-keystore в Git.

После деплоя проверьте URL в `_pushUrl` внутри `index.html`. Запрос без `idToken`, с чужим `chatId` или поддельным `callId` должен возвращать ошибку и не отправлять уведомление.

## 3. Android APK

GitHub Actions запускает проверки и собирает debug APK при push в `main`/`master`, в pull request и вручную. Требования локальной сборки: Node.js 22+ и JDK 21.

```bash
npm ci
npm run verify
npm run test:rules
npm run android:debug
```

Перед сборкой `google-services.json` должен находиться в корне; CI копирует его в `android/app/`. Отладочный ключ не фиксируется в репозитории.

Для публикации в магазине создайте release-keystore отдельно, положите его и пароли в защищённые секреты CI и добавьте release signingConfig. Потеря этого ключа лишит возможности обновлять опубликованное приложение.

## Приёмочная проверка

Нужны два реальных аккаунта и желательно два Android-устройства:

1. регистрация, повторный вход и уникальность `@handle`;
2. личное E2E-сообщение, фото, аудио и ответ на сообщение;
3. невозможность отправки при отсутствии ключа собеседника;
4. создание группы, принятие приглашения, права владельца и участника;
5. аудио- и видеозвонок в активном приложении и при заблокированном экране;
6. кнопки «Принять» и «Отклонить» в нативном экране звонка;
7. контакты, камера, микрофон, геолокация и уведомления;
8. повторная установка собранного APK и запуск без подключения к Google Fonts.

Для звонков между разными мобильными сетями потребуется собственный TURN-сервер. Без TURN WebRTC может не установить соединение за жёстким NAT; STUN не гарантирует связность.
